# AI Debate System - Interview Questions and Answers

This guide answers the most likely technical and system-design questions for the AI Debate System built with FastAPI, LangGraph, Ollama, ChromaDB, WebSockets, and React. The answers are grounded in the current implementation, while also calling out what I would improve in a production version.

## 1. Architecture & System Design

- **Q:** How did you structure the communication between the AI agents in this project? Why did you choose LangGraph over other frameworks like AutoGen or standard LangChain runnables?
  **A:** I structured the debate as an explicit state machine in LangGraph. Each agent is a node, and all of them read and write to a shared `DebateState`. The flow is deterministic: `moderator -> researcher -> pro_agent -> opponent_agent -> fact_checker -> increment_round -> moderator`, with a conditional jump to the `verdict_agent` when the round limit is exceeded. I chose LangGraph because this project is not an open-ended multi-agent conversation; it is a bounded workflow with state, routing, and repeated turns. Compared with plain LangChain runnables, LangGraph gives clearer state transitions and conditional edges. Compared with AutoGen, it gives tighter control over execution order, which matters for a debate pipeline where I want predictable steps like research before argument generation and fact checking after rebuttal.

- **Q:** Explain the flow of data from the moment a user submits a debate topic on the React frontend to the moment the final verdict is displayed.
  **A:** On the frontend, the user enters a topic and submits the form. `handleStart` sends that topic as JSON over a WebSocket connection to FastAPI at `/ws/debate`. The backend creates the initial LangGraph state with fields like `topic`, `current_round`, `conversation`, `fact_checking_results`, and `verdict_data`. It then runs `debate_graph.astream(..., stream_mode="updates")`, which yields state updates as each node finishes. When a node appends a new conversation message, the backend forwards the newest item to the frontend as an `agent_message` event. When the `verdict_agent` returns the final result, the backend sends a separate `verdict` event with both a human-readable summary and a structured scoring payload. The React app listens on `socket.onmessage`, appends each message into `messages`, and renders the final verdict card with scores, confidence, penalties, and rationale.

- **Q:** If we were to scale this debate system to support hundreds of concurrent debates, what architectural bottlenecks would you anticipate, and how would you resolve them?
  **A:** The first bottleneck is model inference. Right now both the argument agents and the verdict agent call a local Ollama model synchronously, so GPU or CPU throughput becomes the main limit. The second bottleneck is that the WebSocket handler is tightly coupled to debate execution, which makes each debate a long-lived connection. The third issue is that ChromaDB and Tavily access happen inline during debate execution, so retrieval latency directly increases end-to-end response time. To scale, I would separate concerns: move model execution into dedicated worker processes or model-serving instances, persist debate state in Redis or Postgres, queue debate jobs, and let the WebSocket layer subscribe to streamed progress. I would also stop clearing and reusing a single knowledge base collection for every debate, because that creates contention and limits multi-tenant behavior. At higher scale I would use a proper vector store with metadata filters, caching, and independent per-session or per-topic namespaces.

- **Q:** How do you handle state across multiple steps of the debate? What happens to the state if the server restarts mid-debate?
  **A:** State is carried in `DebateState`, which includes the topic, round counters, conversation history, fact-check results, search queries, background context, and verdict fields. Each LangGraph node returns a partial update, and reducers merge that update into the shared state. For example, new conversation turns are appended rather than replacing the whole list. In the current version, that state is in memory for the lifetime of the request. If the server restarts mid-debate, the in-flight state is lost and the debate cannot resume. A production fix would be checkpointing state after each node to a durable store, then resuming the graph from the last completed node.

## 2. LangGraph & AI Agents

- **Q:** How do you define and manage state in LangGraph (`DebateState`)? What is the purpose of the `Annotated[..., append_list]` pattern used in your state definition?
  **A:** `DebateState` is a `TypedDict` that defines the full shared state shape for the graph. It includes scalar fields like `topic`, `current_round`, and `background_context`, plus list fields like `conversation`, `fact_checking_results`, and `search_queries`. The `Annotated[..., append_list]` pattern tells LangGraph how to merge updates for list fields. Instead of overwriting the existing list when a node returns a new item, the custom reducer appends the new list onto the previous one. That is important because each node contributes only its own turn or metadata, and the full debate transcript needs to accumulate across the run.

- **Q:** Can you explain how conditional edges work in your graph (`route_from_moderator`)?
  **A:** Conditional edges let the graph choose the next node based on current state rather than a fixed edge. In this project, `route_from_moderator` checks `current_round` and `max_rounds`. If `current_round == 0`, it routes to `researcher` so the debate starts with fresh context gathering. If `current_round > max_rounds`, it routes to `verdict_agent` to end the debate. Otherwise it routes to `pro_agent` to start the next round of argument and rebuttal. This gives the moderator node control over both the initial branch and the termination branch.

- **Q:** How did you design the "Fact Checker" agent? How does it differ from the "Pro" or "Opponent" agents in terms of its system prompts and LLM temperature?
  **A:** The fact checker is intentionally narrower and stricter than the debaters. It takes the latest claim from the conversation, queries the ChromaDB knowledge base for semantically similar facts, combines those retrieved facts with the most recent Tavily-backed web context, and prompts the model to output only `True`, `False`, or `Partially True` plus a one-sentence explanation. It uses `strict_llm` with temperature `0.1`, which reduces creative drift and makes the response format more stable. By contrast, the pro and opponent agents use `base_llm` with temperature `0.7` because they need to generate persuasive arguments, not just constrained judgments.

- **Q:** The researcher agent uses the Tavily API to fetch context. What would you do if the API fails or times out during a live debate?
  **A:** The current implementation already handles failure by catching the exception and storing a fallback string in `background_context`, so the debate can continue in degraded mode instead of crashing. In a production version, I would add explicit request timeouts, retries with backoff, cached results for repeated topics, and a clearer frontend status message that research is unavailable. I would also separate retrieval from the main debate path so one slow dependency does not block the entire turn loop. If the web search still failed, I would proceed with local retrieval only and mark the fact-check confidence as lower.

## 3. WebSockets & FastAPI

- **Q:** Why did you choose WebSockets (`/ws/debate`) instead of standard REST API endpoints (e.g., HTTP POST with Server-Sent Events) for streaming the debate conversation?
  **A:** WebSockets were a good fit because the debate is a long-lived interactive session rather than a single request-response call. Even though the current frontend only sends the topic once, the persistent bidirectional channel makes it easy to add future features like human intervention, moderator overrides, pause and resume, or user follow-up questions without redesigning the transport. SSE could also stream server output for this version, but WebSockets give more flexibility for a multi-turn system where the client may eventually need to send events mid-debate.

- **Q:** How does FastAPI handle concurrent WebSocket connections? Is `asyncio` playing a role here, and how?
  **A:** FastAPI runs the WebSocket endpoint as an async coroutine, so each connection can yield control back to the event loop when it is waiting on network I/O such as `receive_text`, `send_json`, or `asyncio.sleep`. That is where `asyncio` helps concurrency. The caveat is that model inference calls in this code path are not truly async; they are invoked inside graph nodes and can still block if the underlying Ollama calls are slow. So the WebSocket transport is async, but overall concurrency is still constrained by blocking inference work unless that part is moved into separate workers or non-blocking execution.

- **Q:** Suppose a user disconnects their browser mid-debate. How does your backend handle the `WebSocketDisconnect` exception? Does the LangGraph execution continue or stop?
  **A:** In the current implementation, `main.py` catches `WebSocketDisconnect` and logs that the client disconnected. Because the graph streaming loop lives inside the same request handler and there is no detached worker or checkpoint/resume mechanism, the debate effectively stops from the system's point of view. There is no independent job manager that keeps the LangGraph run alive after the socket closes, and there is no recovery path to resume it later.

- **Q:** How do you simulate real-time typing/reading delays before sending the next agent's message to the frontend? (e.g., `await asyncio.sleep(1)`)
  **A:** The backend adds a fixed `await asyncio.sleep(1)` after each streamed update. That is a simple pacing mechanism to keep the UI readable and make the debate feel sequential instead of dumping all messages at once. It is intentionally artificial. If I wanted a more realistic version, I would base the delay on token count, message length, or actual streamed generation timing instead of a fixed one-second pause.

## 4. Local LLMs & RAG (Retrieval-Augmented Generation)

- **Q:** You integrated local models using Ollama (`ChatOllama`). What are the pros and cons of using a local model (like Llama 3) versus a cloud provider (like OpenAI/GPT-4) for this system?
  **A:** The main advantages of a local model are privacy, lower marginal cost per debate, predictable deployment, and the ability to keep both generation and embeddings on-device. That fits well for a demo or an internal tool where you want minimal external data exposure. The tradeoffs are model quality, latency, and throughput. A local model usually gives weaker reasoning and factual reliability than a top-tier cloud model, and scaling local inference for many concurrent debates requires more hardware and operational work. So local models are strong for control and cost, while cloud models are stronger for quality and elastic scaling.

- **Q:** Explain how ChromaDB is used in your Fact Checker agent. What is an embedding function, and why did you use `OllamaEmbeddingFunction`?
  **A:** ChromaDB stores vectorized text snippets that the researcher collects from Tavily. At the start of a debate, the researcher clears the current collection and inserts fresh snippets so the fact checker can query them later. When the fact checker receives a claim, it uses `query_knowledge_base` to retrieve the most relevant snippets by semantic similarity. An embedding function converts raw text into vectors so similarity search becomes possible. I used `OllamaEmbeddingFunction` so embeddings are generated locally through the same Ollama stack, which avoids external embedding providers and avoids Hugging Face model downloads in this setup.

- **Q:** How would you improve the RAG pipeline? For example, if the debate knowledge base grows to millions of documents, how would you ensure the fact-checker remains accurate and fast?
  **A:** I would first change the data model. Right now the knowledge base is cleared per debate, which is fine for a lightweight demo but not for a large corpus. At scale, I would keep a persistent indexed corpus with metadata such as source, topic, timestamp, and trust level. I would use chunking, metadata filters, hybrid retrieval, reranking, and possibly a two-stage pipeline where a fast retriever narrows candidates and a stronger reranker selects the final evidence. I would also add source citations, retrieval evaluation, cache hot queries, and maintain separate namespaces for permanent knowledge versus live session context. That keeps latency low while improving factual grounding.

- **Q:** The Fact Checker combines local ChromaDB facts with recent web context from Tavily. How do you prevent the LLM from hallucinating if the local facts contradict the web search results?
  **A:** The current mitigation is mostly prompt-level: the fact checker is told to answer based only on the supplied facts, and it uses a low-temperature model. That helps, but it is not sufficient for rigorous contradiction handling. A stronger design would pass structured evidence with source attribution, ask the model to explicitly compare conflicting claims, and allow outputs like `Insufficient Evidence` when sources disagree. I would also weight trusted sources more heavily, require citations in the fact-check output, and run a contradiction-detection step before asking the model for a final judgment.

## 5. Frontend & React State Management

- **Q:** How do you maintain the WebSocket connection lifecycle in React? Why is it important to define `socket.onclose` and return `socket.close()` in the `useEffect` cleanup function?
  **A:** The WebSocket is created inside a `useEffect` that runs on mount. Inside that effect, the component registers `onopen`, `onmessage`, and `onclose` handlers, stores the socket in React state, and returns a cleanup function that calls `socket.close()`. The cleanup matters because otherwise you risk leaked connections and duplicate event listeners if the component remounts. Defining `onclose` is also important because it is the natural place to log disconnects, update connection status, or trigger reconnection logic later.

- **Q:** You have a `messages` array in your React state representing the conversation. How do you ensure React correctly updates and re-renders when a new message streams in?
  **A:** Each incoming WebSocket event uses the functional form of state update: `setMessages((prev) => [...prev, newMessage])`. That matters because WebSocket events arrive asynchronously, and the functional form ensures the component always appends to the latest state rather than to a stale closure. Each message also gets a unique `id`, which React uses as the `key` when rendering the message list. Because the state array changes immutably, React detects the update and re-renders the list.

- **Q:** How did you implement the auto-scroll feature (`scrollToBottom`) as new messages arrive? What hooks are involved?
  **A:** I used a `useRef` to point at a sentinel element at the bottom of the message list. Then I used a `useEffect` that depends on `messages`. Every time `messages` changes, the effect calls `scrollIntoView({ behavior: 'smooth' })` on that ref. So the two key hooks are `useRef` for DOM access and `useEffect` for reacting to state changes.

- **Q:** How do you conditionally style the chat bubbles depending on the agent role (e.g., Moderator, Pro, Opponent)?
  **A:** The component uses helper functions like `getRoleTheme`, `getAgentIcon`, and `getAgentName`. `getRoleTheme` returns role-specific class groups for the card background, icon color, label chip, and accent bar. During rendering, each message reads its `role` and applies the corresponding theme. That keeps the styling logic centralized and makes it easy to give each agent a distinct visual identity without duplicating conditional CSS throughout the JSX.

## 6. Behavioral / Scenario-Based

- **Q:** What was the most challenging bug you encountered while building this multi-agent debate system, and how did you debug it?
  **A:** One of the trickier issues was around the verdict step. The verdict agent is asked to return strict JSON, but LLMs sometimes wrap JSON in extra prose or formatting, which can break downstream parsing. There was also a risk of duplicate verdict rendering because the verdict existed both as a conversation message and as a dedicated final event. I debugged that by logging raw model outputs, isolating the exact parse failures, adding `_extract_json_object` to recover valid JSON from noisy output, normalizing the payload before rendering, and guarding against duplicate verdict sends in the WebSocket layer. That bug was a good reminder that multi-agent systems fail as much at interfaces and formatting boundaries as they do in model reasoning.

- **Q:** If you had an extra week to improve this project, what features or technical improvements would you prioritize?
  **A:** My top priority would be persistence and recoverability: store debate state outside memory so debates can survive disconnects or server restarts. Second, I would improve observability and resilience by adding retries, timeout handling, structured logs, and metrics around inference and retrieval latency. Third, I would improve the fact-check pipeline with citations, better contradiction handling, and a more realistic retrieval architecture instead of clearing a shared collection each run. If time remained, I would add user sessions and debate history on the frontend.

- **Q:** How would you implement "human-in-the-loop" so that a real human could override the moderator or jump in as a debater?
  **A:** The cleanest way is to treat the human as another actor in the graph. I would add checkpoints where execution pauses after the moderator, pro, or opponent turn and waits for either a human message or an approval event. The existing WebSocket channel is already a good transport for that because the frontend could send commands like `override_moderator`, `human_reply`, or `resume`. On the backend, I would persist graph state so the system can pause safely, then resume from the correct node once the human input arrives. That would allow hybrid debate modes where the moderator is AI but a human can step in as pro, opponent, or supervisor.
